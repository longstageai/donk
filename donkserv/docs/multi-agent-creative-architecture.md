# 多 Agent 创意目标生成架构设计

## 一、架构定位

现有多 Agent 架构以任务执行链路为核心，主流程为：

```text
目标创意 Agent
  ↓
目标去重 Agent
  ↓
目标价值评审 Agent
  ↓
目标可行性 Agent
  ↓
目标收敛 Agent
  ↓
任务规划 Agent
  ↓
规划审查 Agent
  ↓
任务执行 Agent
  ↓
成果审查 Agent
  ↓
任务交付 Agent
  ↓
任务完成，触发下一轮
```

目标生成层最终输出标准化的 `FinalExecutableGoal`，再交给后续任务规划和执行链路。

## 二、架构设计原则

### 2.1 三层架构

整体架构分为三层：

| 层级 | 核心问题 | 核心产物 |
|---|---|---|
| 目标生成层 | 现在值得让 Agent 做什么？ | FinalExecutableGoal |
| 任务执行准备层 | 这个目标应该如何拆解执行？ | ExecutablePlan |
| 执行交付层 | 如何完成、检查并交付成果？ | FinalDelivery |

### 2.2 Agent 独立存在

所有创意目标相关角色均作为系统一级 Agent 独立存在

系统共包含 10 个独立 Agent：

1. 目标创意 Agent
2. 目标去重 Agent
3. 目标价值评审 Agent
4. 目标可行性 Agent
5. 目标收敛 Agent
6. 任务规划 Agent
7. 规划审查 Agent
8. 任务执行 Agent
9. 成果审查 Agent
10. 任务交付 Agent

### 2.3 最终可执行目标作为规划输入

目标生成层不直接输出执行计划，而是输出最终可执行目标：

```text
FinalExecutableGoal
```

任务规划 Agent 基于 `FinalExecutableGoal` 生成具体执行计划：

```text
FinalExecutableGoal → ExecutablePlan
```

职责边界如下：

```text
目标生成层：决定做什么、为什么做、边界是什么、交付什么
任务执行准备层：决定分几步做、每步用什么工具、依赖关系是什么
执行交付层：执行计划、检查结果、交付成果
```

## 五、分层输入输出边界

### 5.1 目标生成层

#### 输入

```text
核心主题
用户上下文
历史聊天记录
历史已执行任务
桌面向量数据库
用户画像
当前时间
外部参考资料
```

#### 输出

```text
FinalExecutableGoal
```

#### 边界

目标生成层只负责确定目标，不负责拆解具体执行步骤。

它需要明确：

```text
做什么
为什么做
基于什么上下文
目标边界是什么
预期交付物是什么
```

### 5.2 任务执行准备层

#### 输入

```text
FinalExecutableGoal
```

#### 输出

```text
ExecutablePlan
```

#### 边界

任务执行准备层只负责生成可执行计划，不直接执行工具调用。

它需要明确：

```text
执行步骤
工具选择
输入输出
步骤依赖
风险约束
```

### 5.3 执行交付层

#### 输入

```text
ExecutablePlan
```

#### 输出

```text
FinalDelivery
```

#### 边界

执行交付层负责执行计划、检查成果、完成交付和归档。

它需要明确：

```text
执行结果
审查结论
最终成果
交付记录
下一轮触发条件
```

## 六、Agent 职责定义

### 6.1 目标创意 Agent

目标生成链路的起点。

基于核心主题、用户上下文、历史记录、用户画像、当前时间和外部资料，生成一个或多个候选目标。

输入：

```text
核心主题
用户上下文
历史聊天记录
桌面向量数据库
用户画像
当前时间
外部资料
```

输出：

```text
CandidateGoal[]
```

### 6.2 目标去重 Agent

目标生成链路的第一道闸门。

判断候选目标是否与历史已执行任务重复，并拥有一票否决权。

判定结果：

```text
否决：目标与历史任务高度重复，返回目标创意 Agent 重新生成
继续：信息不足，返回目标创意 Agent 补充细节
通过：进入目标价值评审 Agent
```

### 6.3 目标价值评审 Agent

目标质量审查者。

从创造性、差异化、价值密度、聚焦度、主题契合度、个性化程度等维度评审候选目标。

评审维度：

| 维度 | 说明 |
|---|---|
| 创造性 | 是否提出了新颖的思路或方法 |
| 差异化 | 是否与常见同类目标有明显区别 |
| 价值密度 | 是否能在有限执行成本内产生明确价值 |
| 聚焦度 | 是否是单一目标，而不是多个目标混杂 |
| 主题契合度 | 是否符合系统当前核心主题 |
| 个性化程度 | 是否充分利用了用户上下文和用户画像 |

判定结果：

```text
否决：目标价值不足或创意不足，返回目标创意 Agent
继续：目标需要补充或优化，返回目标创意 Agent
通过：进入目标可行性 Agent
```

### 6.4 目标可行性 Agent

Agent 能力边界审查者。

判断目标是否属于 Agent 可直接执行的数字化任务。

检查重点：

```text
是否需要物理操作
是否需要线下行动
是否依赖不可用工具
是否存在权限或安全风险
是否能被后续任务规划 Agent 拆解
是否有明确的预期交付物
```

判定结果：

```text
否决：目标超出 Agent 能力边界，返回目标创意 Agent
继续：目标边界不清，需要调整，返回目标创意 Agent
通过：进入目标收敛 Agent
```

### 6.5 目标收敛 Agent

目标生成层的收尾角色。

整合目标创意 Agent、目标去重 Agent、目标价值评审 Agent、目标可行性 Agent 的输出，生成标准化 `FinalExecutableGoal`。

目标收敛 Agent 不参与评审，只负责：

```text
整理
提炼
结构化输出
```

输出：

```text
FinalExecutableGoal
```

### 6.6 任务规划 Agent

任务执行准备层的核心生产者。

将 `FinalExecutableGoal` 拆解为可执行步骤、工具调用、输入输出和依赖关系。

输出：

```text
ExecutablePlan
```

### 6.7 规划审查 Agent

任务执行准备层的质量控制者。

审查计划是否完整、可行，并且符合 `FinalExecutableGoal` 的目标边界和约束。

判定结果：

```text
不通过：返回任务规划 Agent 重新规划
通过：进入任务执行 Agent
```

### 6.8 任务执行 Agent

真正执行动作的 Agent。

按 `ExecutablePlan` 调用工具，完成每个步骤，并形成执行结果。

输出：

```text
ExecutionResult
```

### 6.9 成果审查 Agent

执行结果质量审查者。

审查 `ExecutionResult` 是否满足 `FinalExecutableGoal` 和 `ExecutablePlan` 的要求。

判定结果：

```text
不通过：返回任务执行 Agent 修正结果
通过：进入任务交付 Agent
```

### 6.10 任务交付 Agent

最终交付和归档 Agent。

格式化成果，选择交付渠道，归档任务，并触发下一轮。

输出：

```text
FinalDelivery
```

## 八、失败回路设计

### 8.1 目标生成层失败回路

```text
目标去重失败 → 回到目标创意 Agent
目标价值评审失败 → 回到目标创意 Agent
目标可行性失败 → 回到目标创意 Agent
```

目标尚未最终确定时，所有问题都回到目标创意 Agent 重新生成、补充或调整。

### 8.2 任务执行准备层失败回路

```text
规划审查失败 → 回到任务规划 Agent
```

目标已经确定后，规划问题优先回到任务规划 Agent 修正。

如果规划审查连续失败，可以升级为：

```text
规划审查 Agent → 目标可行性 Agent
```

重新检查目标边界。

### 8.3 执行交付层失败回路

```text
成果审查失败 → 回到任务执行 Agent
```

计划已经确定后，成果问题优先回到任务执行 Agent 根据反馈修正。

如果连续失败，可以升级为：

```text
成果审查 Agent → 任务规划 Agent
```

重新规划执行步骤。

## 九、与现有架构的关系

新架构不是推翻现有多 Agent 任务执行链路，而是对原有“任务生成 Agent”进行拆分和增强。

原有链路：

```text
任务生成 Agent → 任务规划 Agent → 规划审查 Agent → 任务执行 Agent → 任务审查 Agent → 任务结束 Agent
```

新链路：

```text
目标创意 Agent
→ 目标去重 Agent
→ 目标价值评审 Agent
→ 目标可行性 Agent
→ 目标收敛 Agent
→ 任务规划 Agent
→ 规划审查 Agent
→ 任务执行 Agent
→ 成果审查 Agent
→ 任务交付 Agent
```

其中：

```text
目标创意 Agent
目标去重 Agent
目标价值评审 Agent
目标可行性 Agent
目标收敛 Agent
```

共同替代原有“任务生成 Agent”，并产出 `FinalExecutableGoal`。

后续任务规划、规划审查、任务执行、成果审查和任务交付继续承担原有执行链路职责。

## 十、架构总结

该架构保留了原有多 Agent 系统的主执行链路，同时将目标生成阶段拆解为更细粒度、更可审查、更可控的独立 Agent 链路。

最终系统形成三层结构：

```text
目标生成层：目标创意 → 目标去重 → 目标价值评审 → 目标可行性 → 目标收敛
任务执行准备层：任务规划 → 规划审查
执行交付层：任务执行 → 成果审查 → 任务交付
```

通过该设计，系统可以在任务规划之前先完成目标去重、价值判断和可行性确认，降低重复、空洞、不可执行目标进入执行链路的概率。
